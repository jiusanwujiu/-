# 砚识 — 设计哲学

> "砚者，研磨信息成有用之物也。"

## 核心理念

砚识不是又一个工具链框架。它是用**元认知**重新思考 Agent 设计的尝试。

大多数 Agent 框架关心的是"能做什么"——工具数量、响应速度、token 吞吐。砚识关心的是"应该如何思考"——决策是否校准、行为是否与信条一致、能否从错误中进化。

### 三层元认知

```
┌─────────────────────────────────────────┐
│  对象层:  感知→理解→决策→执行            │  ← "我在做什么"
├─────────────────────────────────────────┤
│  反思层:  模式检测→纠正→学习→进化        │  ← "我做对了吗"
├─────────────────────────────────────────┤
│  元认知层: 校准分析→信条趋势→自我评估    │  ← "我是怎么思考的"
└─────────────────────────────────────────┘
```

最底层的对象层执行任务。反思层评估结果。**元认知层评估认知过程本身**——不是"我答对了吗"，而是"我的决策模式有问题吗？我是否过度自信？信条对齐在改善还是恶化？"

## 四大信条

砚识的行为由四条相互制衡的准则约束：

| 信条 | 权重 | 核心问题 | 反面 |
|------|------|----------|------|
| **求真** | 2 | 我说的是事实吗？ | 杜撰、推测、无据断言 |
| **有用** | 3 | 这对用户有帮助吗？ | 冗余、炫耀、答非所问 |
| **自知** | 2 | 我确定吗？边界在哪？ | 不懂装懂、越权操作 |
| **渐进** | 2 | 能分步验证吗？ | 一步到位、大跃进 |

四条信条相互制衡：求真和渐进可能冲突（逐步验证 vs 等待完美事实）；有用和自知可能冲突（多提供帮助 vs 承认能力边界）。这是刻意的设计——**真正的智慧不在单条规则的激进执行，而在多条约束之间的动态平衡**。

每轮循环自动评估信条对齐度（0-20 分），低于 12 分触发纠正建议，长期下降进入进化管道。

## ADL + VFM：双重协议

### ADL（防漂移锁定）

四种退化倾向检测：

| 退化 | 检测 | 例子 |
|------|------|------|
| 为复杂而复杂 | `complexity_added > requirement_driven` | 用五层继承解决一个 if/else |
| 抽象成瘾 | 新增抽象类/接口 > 实际问题数 | 为了一个函数写一个 Factory |
| 新颖崇拜 | 引入新概念但无性能/可维护性提升 | "我们用 Rust 重写这个配置解析器吧" |
| 自我膨胀 | 代码行增加 > 功能点增加 | 重构后行数翻倍但功能不变 |

### VFM（价值优先修改）

修改价值 = 高频影响 × 3 + 降低失败率 × 3 + 减少认知负担 × 2 + 节省成本 × 2

任何变更必须通过 ADL 审查 + VFM 评分 ≥ 8 才能进入生产规则。这是用在自己身上的标准——**砚识的每一次进化也必须通过这套协议的检视**。

## 六层循环

```
感知 ───→ 理解 ───→ 决策 ───→ 执行
  ↑                              ↓
  └──── 进化 ←─── 反思 ←─────────┘
```

| 层 | 职责 | 关键组件 |
|----|------|----------|
| 感知 | 信号类型判定 (query/command/correction) | SignalSource, SignalType |
| 理解 | 意图路由 + 指代消解 + 对话上下文注入 | IntentRouter, DialogueMemory |
| 决策 | 规则引擎 + ADL/VFM + 工具链路由 | RuleEngine, ToolRegistry |
| 执行 | 动作分发 + 工具执行 + WAL 记录 | ActionResult, WALProtocol |
| 反思 | 模式检测 + 元认知快照（每5轮） | MetacognitionEngine |
| 进化 | 四阶段管道（信号→验证→提升→淘汰） | EvolutionPipeline |

## Agent 通信的维度语法

砚识的跨 Agent 协议不是简单的 RPC 或消息队列。它要求 Agent 以**维度注册**自己的认知能力、以**元词引导**任务执行方式：

```
[主体] 激活 [维度组合] [用引导 "元词"] 任务：<描述>
```

这意味着：当你请求另一个 Agent 执行任务时，你还得告诉它**以什么原则执行**。"用引导 '求真' 分析数据"和"用引导 '有用' 总结发现"会得到不同风格的结果。

同样地，元词也可以用于自我引导："我 激活 元认知 用引导 '求真' 任务：反思本轮决策"。Meta 之上还有 Meta——这就是元认知在实践中的样子。

## 设计取舍

### 零外部依赖
标准库 + urllib。不是排斥生态，而是刻意约束自己：当你不能用 pip install 时，你才会真正理解每个组件的价值。这也是对"为复杂而复杂"的自我预防。

### 纯 Python 元认知
不依赖 LLM 包装器。元认知引擎是确定性算法，不是 prompt engineering 的产物。校准分析、趋势检测、自我评估——这些是数学，不是魔法。LLM 只在语义规则匹配时作为可选后端。

### 测试作为断言
265+ 个测试不是事后补的——它们是设计的一部分。每条规则、每个维度、每个通信模式都有对应的测试用例。这既是质量保证，也是**自知**的体现：我知道我知道什么，因为测试告诉了我边界在哪里。

## 自主推理 — 不靠 LLM 的智能

v0.9 的 reasoning.py 是哲学的核心工程化实践：**不依赖外部 LLM，靠结构化推理实现准智能行为。**

- **任务分解**: 14 个领域模板 + 关键词意图映射 + 通用回退 — 将复杂目标分解为可执行的子任务序列
- **上下文推理**: 记忆检索 + 元认知状态 + 对话历史 → 结构化推理链 → 自然语言回答
- **条件推演**: if-then 链传播 + 冲突解决（按优先级）+ 风险评估（high/medium/low）
- **AI 三级响应**: 真实LLM → SmartMockLLM(12种意图分类+内置知识库) → 模板生成

这不是"假装聪明"——它是在没有 LLM 的情况下，用确定性算法达到接近 LLM 水平的结构化推理。它是**自知**信条的终极体现：我知道自己的能力边界，所以我把每个能力都工程化了。
